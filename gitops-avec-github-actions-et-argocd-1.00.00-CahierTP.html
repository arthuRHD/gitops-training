TP 1 : Git et les concepts de base

Important : les tps sont très scriptés afin que les participants soient autonomes et ne soient jamais bloqués en attendant de l’aide. Cela permet aussi de rejouer tranquillement les scenarios après la formation. Evitez d'utiliser le copier/coller, on retient beaucoup moins qu'en saisissant réellement. Prenez le temps de comprendre les instructions et pourquoi elles sont proposées. Le but n’est évidemment pas de vérifier que les TPs fonctionnent. N’hésitez pas à poser des questions si quelque chose n’est pas clair. Évitez cependant de prendre trop d’avance sur les tps, vous risquez de poser des questions abordées plus tard dans le cours.

Les scripts sont généralement de la forme suivante :

Réaliser une action plus ou moins complexe

détails des actions à effectuer pour réaliser l'action complexe

ou informations pour expliquer l'action

ou remarque particulière

ou encore un bloc de code

commandes à saisir
Sortie attendue des commandes (en gras les éléments notables)

Vous pouvez bien sûr expérimenter mais évitez de trop vous écarter du scénario pour ne pas vous retrouver dans une situation complexe. Idéalement, expérimentez dans un autre dépôt.

Exercice 1 : installation et initialisation

Vérifier si Git est installé

sur Windows, l'explorateur de fichiers montre le menu Open Git Bash here

remarque : dans l'environnement Cloud9, git est déjà installé

on peut aussi vérifier la version installée dans une invite de commandes ou le shell

git version

devrait afficher quelque chose comme git version 2.34.1

Si nécessaire, réaliser l’installation de Git selon votre plateforme

sur Linux : sudo apt install git (ou git-all pour avoir tous les composants)

sur Mac : taper git dans un terminal

sur Windows : télécharger à l’adresse http://git-scm.com/download/win

vérifier le bon fonctionnement

git version

Exercice 2 : disposer d'un compte et créer un dépôt sur GitHub

Se connecter à GitHub

si nécessaire, créer un compte GitHub

il vous faudra confirmer votre inscription (mail de confirmation)

Créer un nouveau dépôt (repository)

indiquer un nom (ex: project01)

cocher les cases Private et Add a README file

cliquer Create repository

le dépôt contient un premier commit avec un fichier README.md

noter le hash court du commit et le message de commit (Initial commit)

observer par ailleur la branche (par défaut main)

Exercice 3 : cloner le dépôt en local

Commencer par récupérer l'url du dépôt (bouton Code)

plusieurs url sont possibles: HTTPS, SSH...

noter que l'url SSH réclame la création d'une clé publique

copier l'URL SSH

Tenter le clone du dépôt et observer un refus

le dépôt est privé, il faut être authentifié (et autorisé)

Ajouter une clé SSH

créer la clé SSH (dans le terminal)

ssh-keygen

afficher la clé publique

cat ~/.ssh/id_rsa.pub

l'ajouter dans GitHub (clic sur l'avatar > Settings > SSH and GPG keys > New SSH key)

Retenter le clone du dépôt en local

observer la récupération des objets git

explorer le répertoire obtenu, le fichier README.md et le dépôt local .git

l'arborescence de fichier est recréée

Tester la commande git remote -v

à quoi correspond origin ?

Solution Exercice 3

l'option -v précise leur url

origin est par convention l'alias indiquant le dépôt d'origine du clone

Exercice 5 : modification et synchronisation

Modifier le fichier README.md

par exemple, ajouter une ligne modified on my desktop

# project01

modified on my desktop

Vérifier le statut avec git status

git status
On branch main
Your branch is up to date with 'origin/main'.

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   README.md

no changes added to commit (use "git add" and/or "git commit -a")

observer les efforts pour indiquer la situation et les indications pour avancer

le fichier README.md est modifié(en rouge)

deux options possibles : add ou restore

observer les différences en effectuant un git diff

git diff
diff --git a/README.md b/README.md
index caef074..4842234 100644
--- a/README.md
+++ b/README.md
@@ -1 +1,3 @@
-# project01
\ No newline at end of file
+# project01
+
+modified on my desktop

noter que la première ligne est aussi modifiée (ajout d'un retour chariot)

Ajouter le fichier README.md dans l'index (staging area) et vérifier à nouveau le statut

git add README.md
git status
On branch main
Your branch is up to date with 'origin/main'.

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        modified:   README.md

cette fois le fichier est modifié (on pourrait encore annuler)

noter qu'on peut aussi ajouter les modifications de tout un répertoire (ex : git add .)

Configurer les préférences

pour effectuer des commits, git nécessite de configurer les user.name et user.email

commencer par afficher les informations actuelles

git config --global --all

corriger si nécessaire

git config --global user.name "votre nom"
git config --global user.email "votre@adresse.mail"

la commande commit utilise aussi un éditeur pour demander un message

configurer l'éditeur de votre choix (exemple pour configurer vim)

git config --global core.editor vim

Effectuer le commit (le système demandera un message de commit)

pour annuler avec vim : [Echap][:][q][!][Entrée]

pour annuler avec nano : [Ctrl-x] puis [n]

git commit
Aborting commit due to empty commit message.

une autre option plus simple est de passer le message sur la ligne de commande

git commit -m "First commit"
[main 9483c14] First commit
 1 file changed, 3 insertions(+), 1 deletion(-)

attention : cette option incite à faire des messages courts, peu explicites

noter le hash court de ce nouveau commit

Vérifier la présence de ce nouveau commit et l'absence de modification en cours

comment ont évolué les références symboliques HEAD et main ?

est-ce que ce commit est présent sur GitHub ?

effectuer l'opération nécessaire pour qu'il soit présent

à quoi correspondent les références origin/main et origin/HEAD ?

Solution Exercice 5

git log --oneline
9483c14 (HEAD -> main) First commit                    ➀  ➁
4b08aa8 (origin/main, origin/HEAD) Initial commit

**git status*
On branch main
Your branch is ahead of 'origin/main' by 1 commit.        ➂
  (use "git push" to publish your local commits)

nothing to commit, working tree clean

git push                                                ➃
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Writing objects: 100% (3/3)...
...

git log --oneline
9483c14 (HEAD -> main, origin/main, origin/HEAD) First commit ➄
4b08aa8 Initial commit

git status
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean

le commit est bien apparu en tête de log ➀

HEAD référence toujours main comme l'indique HEAD -> main ➁

mais main a évolué et référence maintenant le nouveau commit

le git status indique que la branche locale main est maintenant en tête devant origin/main ➂

le commit est local, pour le pousser vers le dépôt distant, il faut un git push ➃

après le git push, la situation a changé : le dépôt local et le dépôt distant sont synchronisés

les références origin/HEAD et origin/main pointent le dernier état connu du dépôt distant

git push a poussé le commit sur le dépôt distant et a resynchonisé ce dernier état connu ➁ ➄

Exercice 6 : désynchronisation et historique

Créer une nouvelle modification du fichier README.md sur GitHub cette fois

modifier le fichier dans l'interface web (ajouter une ligne)

effectuer le commit (un message standard Update README.md est proposé)

Revenir sur le terminal et vérifier le status

git status
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean

la copie locale n'est pas au courant des évolutions sur le dépôt distant

le contenu du fichier README.md en local n'est pas à jour

en effet, Git fonctionne de manière déconnectée, il faut explicitement se resynchroniser

Effectuer un git fetch pour ramener les nouveaux commits en local

git fetch
remote: Enumerating objects: 5, done.
remote: Counting objects: 100% (5/5), done.
remote: Compressing objects: 100% (2/2), done.
remote: Total 3 (delta 1), reused 0 (delta 0), pack-reused 0
Unpacking objects: 100% (3/3), 912 bytes | 57.00 KiB/s, done.
From https://github.com/<votre.nom>/project01
   5135700..1e450db  main       -> origin/main

le commit est détecté et récupéré en local

git status et git log nous montrent que la situation a changée

git status
On branch main
Your branch is behind 'origin/main' by 1 commit, and can be fast-forwarded.
  (use "git pull" to update your local branch)

nothing to commit, working tree clean

git log --oneline
9483c14 (HEAD -> main) First commit
4b08aa8 Initial commit

git fetch a ramené le commit et resynchonisé ses références distantes

origin/main et origin/HEAD ne sont plus présentes sur les commits listés

notre branche main locale est maintenant en retard (behind origin/main)

pourtant le commit est bien là, git log --all permet de le voir

git log --oneline --all
5135700 (origin/main, origin/HEAD) Update README.md
9483c14 (HEAD -> main) First commit
4b08aa8 Initial commit

Observer le contenu actuel du fichier README.md

le fichier n'est pas encore à jour

git fetch ne touche pas à la working copy

pour cela il faut effectuer le git pull mentionné dans le status

Effectuer le git pull

git pull
Updating 9483c14..5135700
Fast-forward
 README.md | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)

observer le contenu du README.md ⇒ il est maintenant à jour

vérifier le git log

git log --oneline
5135700 (HEAD -> main, origin/main, origin/HEAD) Update README.md
9483c14 First commit
4b08aa8 Initial commit

le Fast-forward a consisté à repositionner HEAD (ou plutôt main)

Effectuer un git checkout 9483c14 ⇐ mettre ici le hash du second commit

noter l'avertissement sans trop s'y attarder

observer le contenu du README.md revenu à l'état précédent

observer l'affichage d'un git log --oneline puis d'un git log --oneline --all

recommencer avec un git checkout 4b08aa8

à quoi correspond la référence symbolique HEAD ?

que signifie le detached HEAD state de l'avertissement ?

revenir à un état normal à jour (où HEAD -> main)

Solution Exercice 6

git checkout 9483c14 ⇐ mettre ici le bon hash
Note: switching to '9483c14'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by switching back to a branch.

If you want to create a new branch to retain commits you create, you may
do so (now or later) by using -c with the switch command. Example:

  git switch -c <new-branch-name>

Or undo this operation with:

  git switch -

Turn off this advice by setting config variable advice.detachedHead to false

HEAD is now at 9483c14 First commit

cat README.md
# project01

modified on my desktop

git log --oneline
9483c14 (HEAD) First commit
4b08aa8 Initial commit

git log --oneline --all
5135700 (origin/main, origin/HEAD, main) Update README.md
9483c14 (HEAD) First commit
4b08aa8 Initial commit

le HEAD est détaché de la branche main

il indique le commit de référence pour la working copy

git checkout 4b08aa8 ⇐ mettre ici le hash du tout premier commit
Previous HEAD position was 9483c14 First commit
HEAD is now at 4b08aa8 Initial commit

cat README.md
# project01

git log --oneline
4b08aa8 (HEAD) Initial commit

git log --oneline --all
5135700 (origin/main, origin/HEAD, main) Update README.md
9483c14 First commit
4b08aa8 (HEAD) Initial commit

git checkout permet de repositionner le HEAD et les fichiers de la working copy

se rappeler que git log est en réalité équivalent à git log HEAD

il montre la chaine d'historique en partant du commit courant, pointé par HEAD

git log main ou bien git log 5135700 part d'un autre commit

et git log --all montre tous les commits

git checkout main
Previous HEAD position was 4b08aa8 Initial commit
Switched to branch 'main'
Your branch is up to date with 'origin/main'.

git log --oneline
1e450db (HEAD -> main, origin/main, origin/HEAD) Update README.md
5135700 Update README.md
9483c14 First commit
4b08aa8 Initial commit

git checkout main (ou git switch main) réattache le HEAD à la branche main

il ne pointe plus directement un commit mais pointe une branche

et remet l'état de la working copy à jour

TP 2 : Git et les branches

Exercice 1 : manipuler les branches

Afficher les branches

git branch
* main

git branch --all
* main
  remotes/origin/HEAD -> origin/main
  remotes/origin/main

noter l'astérisque indiquant la branche est active

l'option --all montre aussi la connaissance locale des branches distantes

Effectuer un checkout sur le commit initial

adapter bien sûr avec le hash du premier commit (git log pour le retrouver)

git checkout 4b08aa8

comme précédemment, le HEAD est détaché

et le fichier README.md est revenu à son état initial

Créer et switcher sur une branche-a à partir de ce point

git switch -c branche-a

Afficher à nouveau les branches et le log

git branch
* branche-a
  main 

git log
commit 4b08aa87591e53336542bbb08766f62c3b6d360d (HEAD -> branche-a)
...

la branche-a a été créée (option -c)

elle est maintenant la branche active (à cause du switch)

le log montre le même commit initial

le HEAD pointe maintenant la branche-a

Modifier le fichier README.md et effectuer un commit

ajouter simplement une ligne Ajout dans la branche-a

peut se faire avec un éditeur ou en ligne de commande

echo "ajout dans la branche-a" >> README.md

git add README.md

git commit -m "Ajoute la feature-a"

git log --oneline
b346f5b (HEAD -> branche-a) Ajoute la feature-a
4b08aa8 Initial commit

ici encore, un commit est créé avec le commit pointé par HEAD comme parent

la branche courante (active) est mise à jour et pointe ce nouveau commit

Ajouter un fichier feature-a.txt et le commiter

echo "code de la feature-a" > feature-a.txt

git add feature-a.txt

git commit -m "Finalise la feature-a"

Observer un log graphique (option --graph de toutes (--all) les branches

git log --oneline --all --graph
* 9024011 (HEAD -> branche-a) Finalise la feature-a
* b346f5b Ajoute la feature-a
| * 1e450db (origin/main, origin/HEAD, main) Update README.md
| * 5135700 Update README.md
| * 9483c14 First commit
|/  
* 4b08aa8 Initial commit

noter que nous avons rétrospectivement simulé un développement parallèle

remarque : les astérisques indiquent à quelle branche appartient le commit de la ligne

la branche de gauche est la branche branche-a

la branche de droite est la branche main contenant les 3 commits précédents

le commit 4b08aa8 est celui où les branches ont divergées

tout se passe comme si la branche-a était partie avant que main évolue

Exercice 2 : merger les branches

Le développement de la feature-a dans la branche-a est terminé. Noter qu'il peut correspondre à tout développement (exploration, feature, développement en parallèle, hotfix, ticket JIRA...). Nous souhaitons maintenant incorporer ces modifications dans la branche principale, c'est à dire merger.

Basculer sur la branche main et observer le contenu du répertoire

git switch main
Switched to branch 'main'
Your branch is up to date with 'origin/main'.

ls
README.md

cat README.md
# project01

modified on my desktop
modified on github

le fichier feature-a.txt a disparu

le contenu du README.md est revenu à l'état main

remarque : les branches sont isolées, passer de l'une à l'autre est très rapide

Déclencher un merge pour ramener la branche-a vers la branche courante

git merge branche-a
Auto-merging README.md
CONFLICT (content): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.

noter que le merge automatique aboutit à un conflit dans le fichier README.md

remarque : le prompt rappele qu'un merge est en cours

Observer le contenu du répertoire et du fichier README.md

ls 
README.md  feature-a.txt

cat README.md 
# project01
<<<<<<< HEAD

modified on my desktop
modified on github
=======
ajout dans la branche-a
>>>>>>> branche-a

le fichier feature.txt a été mergé automatiquement

un conflit est marqué dans le fichier README.md

remarque : la première ligne n'est pas en conflit (existait avant branche-a)

Résoudre le conflit

corriger manuellement le fichier avec un éditeur (exemple garder les lignes des deux branches)

attention à bien enlever les lignes de marquages (chevrons)

# project01

modified on my desktop
modified on github 
ajout dans la branche-a

il faut alors prendre en compte ces nouvelles modifications (add, status...)

et terminer le merge

git status
...
git add README.md
git merge --continue

Solution Exercice 3

Un tag est fait pour être figé, au contraire d'une branche qui évolue à chaque commit

On peut bien sûr créer plusieurs tags sur le même commit

Lorsqu'on pousse avec git push, il ne se passe rien

les tags n'apparaissent pas sur le serveur

La documentation indique l'option --tags pour pousser tous les tags

remarque : c'est généralement déconseillé de pousser tous les tags

on risque de repousser des tags supprimés

il est préférable de pousser chaque tag créé avec git push origin <tag>

les tags poussés apparaissent bien sur le serveur

Créer les fichiers suivants :

deployment.yaml dans helm-tpl/templates

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: clinic-deploy
  labels:
    myversion: "1"
spec:
  replicas: {{ .Values.replicas }}
  selector:
    matchLabels:
      app: clinic-front
  template:
    metadata:
      labels:
        app: clinic-front
    spec: # specification du pod
      containers:
        - name: springboot
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"

          startupProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            timeoutSeconds: 2
            periodSeconds: 3
            initialDelaySeconds: 2
            failureThreshold: 10

          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            timeoutSeconds: 1
            periodSeconds: 3

          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            timeoutSeconds: 1
            periodSeconds: 3

service.yaml dans helm-tpl/templates

---
kind: Service
apiVersion: v1

  name: clinic-service

  selector:
    app: clinic-front
  ports:
  # Default port used by the image
  - port: 8080

ingress.yaml dans helm-tpl/templates

---

  name: clinic-ingress

  rules:
  - http:
      paths:
      - pathType: Prefix
        path: /

            name: clinic-service
            port:
              number: 8080

Chart.yaml dans helm-tpl

apiVersion: v2
name: petclinic
description: A Helm chart for Pet Clinic

# A chart can be either an 'application' or a 'library' chart.
#
# Application charts are a collection of templates that can be packaged into versioned archives
# to be deployed.
#
# Library charts provide useful utilities or functions for the chart developer. They're included as
# a dependency of application charts to inject those utilities and functions into the rendering
# pipeline. Library charts do not define any templates and therefore cannot be deployed.
type: application

# This is the chart version. This version number should be incremented each time you make changes
# to the chart and its templates, including the app version.
# Versions are expected to follow Semantic Versioning (https://semver.org/)
version: 0.1.1

# This is the version number of the application being deployed. This version number should be
# incremented each time you make changes to the application. Versions are not expected to
# follow Semantic Versioning. They should reflect the version the application is using.
# It is recommended to use it with quotes.
appVersion: "3.2.0-SNAPSHOT"

values.yaml dans helm-tpl

image:
  repository: "spring-petclinic"

replicas: 1

values_pp.yaml dans helm-tpl

replicas: 2

image:
  tag: 3.2.1-SNAPSHOT
